# 第3回: テーブル設計 (Table Design)

## 1. DDL — データ定義言語 (Data Definition Language)

### 1.1 CREATE TABLE

```sql
CREATE TABLE products (
    product_id   INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name         TEXT NOT NULL,
    price        NUMERIC(10, 2) NOT NULL CHECK (price >= 0),
    category_id  INTEGER REFERENCES categories(category_id),
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### 1.2 ALTER TABLE

テーブル構造の変更は `ALTER TABLE` で行う。

```sql
-- カラム追加 (ADD COLUMN)
ALTER TABLE products ADD COLUMN description TEXT;

-- カラム削除 (DROP COLUMN)
ALTER TABLE products DROP COLUMN description;

-- カラム名変更 (RENAME COLUMN)
ALTER TABLE products RENAME COLUMN name TO product_name;

-- データ型変更 (ALTER TYPE)
ALTER TABLE products ALTER COLUMN product_name TYPE VARCHAR(200);

-- デフォルト値の設定・削除
ALTER TABLE products ALTER COLUMN price SET DEFAULT 0;
ALTER TABLE products ALTER COLUMN price DROP DEFAULT;

-- NOT NULL制約の追加・削除
ALTER TABLE products ALTER COLUMN description SET NOT NULL;
ALTER TABLE products ALTER COLUMN description DROP NOT NULL;

-- 制約の追加
ALTER TABLE products ADD CONSTRAINT price_positive CHECK (price > 0);

-- 制約の削除
ALTER TABLE products DROP CONSTRAINT price_positive;
```

**注意**: 本番環境での `ALTER TABLE` は、テーブルサイズやロック挙動を考慮する必要がある。`ADD COLUMN` に `DEFAULT` を付けると PostgreSQL 11+ ではテーブル書き換えが発生しない（メタデータのみ更新）。一方、`ALTER TYPE` は全行の書き換えが必要になるケースが多い。

### 1.3 DROP TABLE / TRUNCATE

```sql
-- テーブル削除（依存オブジェクトも含めて削除）
DROP TABLE IF EXISTS products CASCADE;

-- データのみ全削除（構造は残す / DELETE より高速）
TRUNCATE TABLE products RESTART IDENTITY CASCADE;
```

`TRUNCATE` は内部的にテーブルファイルを切り捨てるため `DELETE` より高速だが、行単位のトリガは発火しない。トランザクション内で実行可能（ロールバック可能）。

## 2. データ型の選択指針 (Data Type Selection)

### 2.1 識別子 (Identifiers)

| 選択肢 | 推奨場面 |
|--------|---------|
| `SERIAL` / `BIGSERIAL` | レガシー互換。内部的にシーケンスを作る |
| `GENERATED ALWAYS AS IDENTITY` | **SQL標準準拠、推奨**。値の手動挿入を防止 |
| `GENERATED BY DEFAULT AS IDENTITY` | 手動挿入も許可したい場合 |
| `UUID` | 分散システム、外部公開ID。`gen_random_uuid()` で生成 |

```sql
-- 推奨: IDENTITY
CREATE TABLE users (
    user_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY
);

-- UUID（外部公開ID用）
CREATE TABLE api_keys (
    key_id    UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id   INTEGER NOT NULL REFERENCES users(user_id)
);
```

`SERIAL` vs `IDENTITY`: `SERIAL` は PostgreSQL 独自構文でシーケンスを暗黙的に作成する。`IDENTITY` は SQL 標準であり、`GENERATED ALWAYS` にすれば `INSERT` 時の手動値指定を拒否できるため、より安全。

### 2.2 文字列型

| 型 | 特徴 |
|----|------|
| `TEXT` | 可変長、長さ制限なし。**PostgreSQL では `VARCHAR` と内部的に同等** |
| `VARCHAR(n)` | 最大 n 文字。アプリ層で長さを検証するなら不要 |
| `CHAR(n)` | 固定長。パディングされる。ほぼ使わない |

**PostgreSQL での推奨**: 特段の理由がなければ `TEXT` を使い、長さ制約が必要なら `CHECK` 制約で表現する。`TEXT` と `VARCHAR` に性能差はない。

```sql
-- TEXT + CHECK で長さ制限
CREATE TABLE users (
    email TEXT NOT NULL CHECK (char_length(email) <= 254)
);
```

### 2.3 日時型

| 型 | 特徴 |
|----|------|
| `TIMESTAMP` | タイムゾーンなし。格納値をそのまま返す |
| `TIMESTAMPTZ` | **推奨**。格納時にUTCへ変換、取得時にセッションTZで表示 |
| `DATE` | 日付のみ |
| `INTERVAL` | 期間 |

**原則**: `TIMESTAMPTZ` を使う。`TIMESTAMP`（タイムゾーンなし）はタイムゾーンの異なる環境でバグの温床になる。

### 2.4 数値型

| 型 | 特徴 |
|----|------|
| `INTEGER` / `BIGINT` | 整数。4バイト / 8バイト |
| `NUMERIC(p, s)` | 任意精度の10進数。**金額計算に必須** |
| `REAL` / `DOUBLE PRECISION` | 浮動小数点（IEEE 754）。丸め誤差あり |

```sql
-- 金額は NUMERIC
CREATE TABLE order_items (
    unit_price NUMERIC(10, 2) NOT NULL,
    quantity   INTEGER NOT NULL CHECK (quantity > 0)
);

-- 科学計算・機械学習の特徴量は DOUBLE PRECISION
CREATE TABLE measurements (
    value DOUBLE PRECISION NOT NULL
);
```

### 2.5 その他

| 型 | 用途 |
|----|------|
| `BOOLEAN` | 真偽値 |
| `JSONB` | 半構造化データ（GINインデックス対応） |
| `ARRAY` | 配列型（正規化できない場合に限定使用） |
| `INET` / `CIDR` | IPアドレス |
| `BYTEA` | バイナリデータ（大きなファイルは外部ストレージ推奨） |

## 3. 制約 (Constraints)

制約はデータの整合性をデータベースレベルで保証する仕組みである。アプリ層のバリデーションだけに頼ると、直接SQLでデータを修正した場合やバグにより不正データが入る。

### 3.1 PRIMARY KEY

主キー。`NOT NULL` + `UNIQUE` と等価だが、テーブルに1つだけ定義できる。

```sql
-- 単一カラム
CREATE TABLE categories (
    category_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name        TEXT NOT NULL
);

-- 複合主キー (Composite Primary Key)
CREATE TABLE order_items (
    order_id   INTEGER REFERENCES orders(order_id),
    product_id INTEGER REFERENCES products(product_id),
    quantity   INTEGER NOT NULL CHECK (quantity > 0),
    PRIMARY KEY (order_id, product_id)
);
```

### 3.2 FOREIGN KEY

外部キーは参照整合性 (referential integrity) を保証する。参照先の行が削除・更新されたときの挙動を指定できる。

```sql
CREATE TABLE orders (
    order_id    INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    user_id     INTEGER NOT NULL REFERENCES users(user_id) ON DELETE RESTRICT,
    ordered_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

| アクション | 意味 |
|-----------|------|
| `RESTRICT` | 参照している行がある場合、親の削除を拒否（デフォルト） |
| `CASCADE` | 親を削除すると子も連鎖削除。**危険なので慎重に使う** |
| `SET NULL` | 親を削除すると子のFKカラムを `NULL` にする |
| `SET DEFAULT` | 親を削除すると子のFKカラムをデフォルト値にする |
| `NO ACTION` | `RESTRICT` と類似だがチェックのタイミングが異なる（遅延可能） |

**使い分けの指針**:
- `CASCADE`: ログ・履歴など、親に従属するデータ（例: ユーザー削除時にセッション情報も削除）
- `SET NULL`: 関連が切れても子のデータは残すべき場合（例: 担当者が退職しても案件は残す）
- `RESTRICT`: 参照先が消えてはならない場合（例: 注文に紐づく商品は削除不可）

### 3.3 UNIQUE

```sql
CREATE TABLE users (
    user_id  INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    email    TEXT NOT NULL,
    UNIQUE (email)
);

-- 複数カラムの組み合わせにUNIQUE
ALTER TABLE enrollments ADD CONSTRAINT unique_enrollment
    UNIQUE (student_id, course_id);
```

**NULL の扱い**: PostgreSQL では `UNIQUE` 制約に対して複数の `NULL` を許可する（NULL ≠ NULL のため）。PostgreSQL 15+ では `NULLS NOT DISTINCT` を指定して NULL も一意制約の対象にできる。

### 3.4 CHECK

```sql
CREATE TABLE products (
    price    NUMERIC(10, 2) CHECK (price >= 0),
    discount NUMERIC(3, 2)  CHECK (discount BETWEEN 0 AND 1),
    -- 複数カラムにまたがるCHECK
    CONSTRAINT valid_dates CHECK (end_date > start_date)
);
```

### 3.5 NOT NULL / DEFAULT

```sql
CREATE TABLE audit_log (
    log_id     INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    action     TEXT NOT NULL,
    details    JSONB NOT NULL DEFAULT '{}',
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**設計指針**: 原則として `NOT NULL` を付け、NULL を許可する場合は明確な理由を持つ（例: 「まだ未定」を表現する必要がある）。

## 4. 正規化 (Normalization)

正規化とは、データの冗長性を排除し更新異常 (update anomaly)、挿入異常 (insertion anomaly)、削除異常 (deletion anomaly) を防ぐためのテーブル分割手法である。

### 4.1 非正規形の例: EC注文テーブル

以下のテーブルは複数の異常を抱えている。

```
orders_raw
| order_id | customer_name | customer_email     | product_names          | prices      | category   |
|----------|---------------|--------------------|------------------------|-------------|------------|
| 1        | 田中太郎      | tanaka@example.com | ノートPC, マウス        | 80000, 3000 | 電子機器   |
| 2        | 佐藤花子      | sato@example.com   | ノートPC               | 80000       | 電子機器   |
| 3        | 田中太郎      | tanaka@example.com | キーボード             | 5000        | 電子機器   |
```

**問題点**:
- **更新異常**: 田中太郎のメールアドレスを変えるには2行を更新する必要がある。1行だけ更新すると不整合
- **挿入異常**: 注文なしで新商品を登録できない（商品が注文に従属している）
- **削除異常**: 佐藤花子の注文を削除すると、佐藤花子の顧客情報も失われる
- `product_names` と `prices` が複数値を含む（繰り返しグループ）

### 4.2 第一正規形 (1NF)

**条件**: すべてのカラムが原子値 (atomic value) を持つ。繰り返しグループがない。

```
orders_1nf
| order_id | customer_name | customer_email     | product_name | price | category |
|----------|---------------|--------------------|--------------|-------|----------|
| 1        | 田中太郎      | tanaka@example.com | ノートPC      | 80000 | 電子機器 |
| 1        | 田中太郎      | tanaka@example.com | マウス        | 3000  | 電子機器 |
| 2        | 佐藤花子      | sato@example.com   | ノートPC      | 80000 | 電子機器 |
| 3        | 田中太郎      | tanaka@example.com | キーボード    | 5000  | 電子機器 |
```

繰り返しグループを展開した。しかし冗長性は残っている。

### 4.3 第二正規形 (2NF)

**条件**: 1NF であり、非キー属性がすべて候補キー全体に完全関数従属 (full functional dependency) している。

1NFテーブルの候補キーは `{order_id, product_name}` だが、`customer_name` と `customer_email` は `order_id` のみに依存している（部分関数従属 / partial dependency）。

分割:

```sql
-- orders: order_id → customer_name, customer_email
CREATE TABLE orders (
    order_id       INTEGER PRIMARY KEY,
    customer_name  TEXT NOT NULL,
    customer_email TEXT NOT NULL
);

-- order_items: {order_id, product_name} → price, category
CREATE TABLE order_items (
    order_id     INTEGER REFERENCES orders(order_id),
    product_name TEXT NOT NULL,
    price        NUMERIC(10,2) NOT NULL,
    category     TEXT NOT NULL,
    PRIMARY KEY (order_id, product_name)
);
```

### 4.4 第三正規形 (3NF)

**条件**: 2NF であり、非キー属性間の推移的関数従属 (transitive dependency) がない。

`order_items` では `product_name → category` という推移的従属がある（カテゴリは商品で決まり、注文には依存しない）。

分割:

```sql
CREATE TABLE products (
    product_id  INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name        TEXT NOT NULL UNIQUE,
    price       NUMERIC(10,2) NOT NULL,
    category_id INTEGER REFERENCES categories(category_id)
);

CREATE TABLE categories (
    category_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name        TEXT NOT NULL UNIQUE
);

CREATE TABLE orders (
    order_id       INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    customer_id    INTEGER NOT NULL REFERENCES customers(customer_id),
    ordered_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE customers (
    customer_id INTEGER GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    name        TEXT NOT NULL,
    email       TEXT NOT NULL UNIQUE
);

CREATE TABLE order_items (
    order_id   INTEGER REFERENCES orders(order_id),
    product_id INTEGER REFERENCES products(product_id),
    quantity   INTEGER NOT NULL CHECK (quantity > 0),
    unit_price NUMERIC(10,2) NOT NULL,  -- 注文時点の価格を記録
    PRIMARY KEY (order_id, product_id)
);
```

**注意**: `order_items.unit_price` は `products.price` と重複に見えるが、注文時点の価格を保持するために必要。これは正規化違反ではなく、意図的な設計である。

### 4.5 ボイス・コッド正規形 (BCNF)

**条件**: すべての非自明な関数従属 `X → Y` において、X が候補キー（superkey）である。

3NF と BCNF の差が出るのは、候補キーが複数あり重複するカラムを含む場合に限られる。実務では 3NF まで正規化すれば大半のケースで十分。

例: 授業スケジュール

```
schedule(student_id, subject, instructor)
```

関数従属:
- `{student_id, subject} → instructor`（学生と科目で講師が決まる）
- `instructor → subject`（講師は1科目のみ担当）

`instructor → subject` の左辺 `instructor` は候補キーではないため、BCNF に違反。

分割:

```sql
CREATE TABLE instructor_subjects (
    instructor_id INTEGER PRIMARY KEY,
    subject       TEXT NOT NULL
);

CREATE TABLE enrollments (
    student_id    INTEGER,
    instructor_id INTEGER REFERENCES instructor_subjects(instructor_id),
    PRIMARY KEY (student_id, instructor_id)
);
```

### 4.6 反正規化 (Denormalization) のトレードオフ

正規化は整合性を高めるが、JOIN が増えて読み取り性能が低下する場合がある。以下のケースでは意図的な反正規化を検討する。

| ケース | 手法 | 例 |
|--------|------|----|
| 集計値のキャッシュ | 集計カラムの追加 | `orders.total_amount` |
| 読み取り最適化 | マテリアライズドビュー | 商品一覧 + カテゴリ名 |
| 履歴の保持 | スナップショットの埋め込み | `order_items.unit_price` |
| 頻繁な JOIN の排除 | 冗長カラムの追加 | `order_items.product_name` |

**反正規化のルール**:
1. まず正規化する。性能問題が実測されてから反正規化を検討する
2. 反正規化した箇所は文書化する
3. データの整合性をトリガやアプリ層で担保する仕組みを用意する

## 5. インデックス (Indexes)

インデックスはテーブル全体をスキャンせずに目的の行を高速に見つけるためのデータ構造。ただし書き込み時のオーバーヘッドとストレージコストが発生する。

### 5.1 B-tree インデックス（デフォルト）

PostgreSQL で `CREATE INDEX` を実行すると B-tree が作られる。

```sql
CREATE INDEX idx_products_name ON products (name);
```

**B-tree の内部構造**:

```
          [M]              ← ルートノード
         /   \
      [D,H]  [R,V]        ← 内部ノード (ブランチ)
      / | \   / | \
   [A-C][D-G][H-K] ...    ← リーフノード（実際のデータへのポインタ）
```

- バランス木であり、どのリーフノードまでも同じ深さ（$O(\log n)$ で探索）
- リーフノードは双方向リンクリストで連結 → 範囲検索が効率的
- 等値比較 (`=`)、範囲比較 (`<`, `>`, `BETWEEN`)、`ORDER BY`、`IS NULL` に対応
- PostgreSQL の B-tree はリーフに heap tuple の TID（行の物理位置）を格納

### 5.2 Hash インデックス

等値比較 (`=`) のみに特化。範囲検索は不可。

```sql
CREATE INDEX idx_users_email_hash ON users USING hash (email);
```

PostgreSQL 10 以降で WAL 対応（crash-safe）になったが、B-tree が等値比較にも対応するため、Hash の出番は限定的。超大量のユニークキーに対する完全一致検索でわずかに有利な場合がある。

### 5.3 GIN インデックス (Generalized Inverted Index)

複数の値を含むデータ型に適する。転置インデックスの構造。

```sql
-- JSONB
CREATE INDEX idx_products_attrs ON products USING gin (attributes);
-- → WHERE attributes @> '{"color": "red"}' が高速化

-- 配列
CREATE INDEX idx_tags ON articles USING gin (tags);
-- → WHERE tags @> ARRAY['postgresql'] が高速化

-- 全文検索 (Full-Text Search)
CREATE INDEX idx_content_fts ON articles USING gin (to_tsvector('japanese', content));
-- → WHERE to_tsvector('japanese', content) @@ to_tsquery('PostgreSQL') が高速化
```

### 5.4 GiST インデックス (Generalized Search Tree)

幾何データ、範囲型、全文検索など多様なデータ型をサポート。

```sql
-- 地理データ（PostGIS）
CREATE INDEX idx_locations_geom ON locations USING gist (geom);

-- 範囲型の重なり検索
CREATE INDEX idx_reservations_period ON reservations USING gist (period);
-- → WHERE period && '[2024-01-01, 2024-01-31]'::daterange
```

### 5.5 複合インデックスとカラム順序

複合インデックス (composite index) はカラムの順序が重要。B-tree は左端のカラムから順にソートされる。

```sql
CREATE INDEX idx_orders_user_date ON orders (user_id, ordered_at);
```

このインデックスが有効なクエリ:

```sql
-- ✓ 有効: 左端カラムを使用
WHERE user_id = 100

-- ✓ 有効: 左端 + 2番目
WHERE user_id = 100 AND ordered_at > '2024-01-01'

-- ✗ 無効: 左端カラムなし
WHERE ordered_at > '2024-01-01'
```

**設計原則**:
1. 等値条件 (`=`) のカラムを先に置く
2. 範囲条件 (`<`, `>`, `BETWEEN`) のカラムを後に置く
3. よく使われる `WHERE` 条件のカラム組み合わせから決める

### 5.6 部分インデックス (Partial Index)

条件に合致する行のみをインデックス化する。インデックスサイズの削減と性能向上が期待できる。

```sql
-- 未発送の注文だけをインデックス化
CREATE INDEX idx_orders_pending ON orders (ordered_at)
    WHERE status = 'pending';

-- アクティブユーザーだけ
CREATE INDEX idx_users_active ON users (email)
    WHERE is_active = true;
```

### 5.7 式インデックス (Expression Index)

カラム値を加工した結果にインデックスを作成する。

```sql
-- 大文字小文字を区別しない検索用
CREATE INDEX idx_users_email_lower ON users (lower(email));
-- → WHERE lower(email) = 'user@example.com' で使われる

-- 日付の年だけで検索
CREATE INDEX idx_orders_year ON orders (extract(year FROM ordered_at));
```

**注意**: クエリの `WHERE` 句で使う式とインデックスの式が完全に一致する必要がある。

### 5.8 インデックスを張るべきでない場合

| 状況 | 理由 |
|------|------|
| テーブルが小さい（数千行以下） | シーケンシャルスキャンの方が速い |
| カーディナリティが低い（例: `BOOLEAN`、`status` が3種類） | 大量の行がヒットし、インデックスの効果が薄い |
| 書き込みが極めて多い | INSERT/UPDATE/DELETE の都度インデックスも更新される |
| ほぼ使われないクエリのため | ストレージとメンテナンスコストが無駄 |

**カーディナリティの目安**: テーブル行数の 10–15% 以上がヒットするクエリでは、プランナがインデックスより Seq Scan を選ぶことが多い。

## まとめ: テーブル設計チェックリスト

1. **データ型**: 目的に合った最小のデータ型を選ぶ。金額は `NUMERIC`、時刻は `TIMESTAMPTZ`
2. **制約**: `NOT NULL` を原則とし、PK/FK/UNIQUE/CHECK で整合性を保証する
3. **正規化**: まず 3NF にする。性能問題が出たら反正規化を検討する
4. **インデックス**: よく使われるクエリの `WHERE`/`JOIN`/`ORDER BY` に合わせて設計する
5. **命名規則**: 一貫したネーミング（例: `テーブル名_カラム名` で制約名を付ける）を維持する
